## How to check progress mid-run

```bash
RUN=$(ls -t /home/opc/202603-shopdatav2/data/soak/*.status | head -1)
cat "$RUN"                        # STARTING / RUNNING / SUMMARIZING / COMPLETED / FAILED-*
wc -l "${RUN%.status}.jsonl"      # how many fetches done
tail -3 "${RUN%.status}.jsonl"    # latest results
```

The runner re-runs `soak_summarize.py` after every fetch, so
`docs/soak/100-asin-24h-2026-05-01.md` is also kept fresh — `cat`-ing
this file at any time during the 24h gives a current snapshot.

## How to abort cleanly

```bash
ps -ef | grep soak_100_asin_24h.sh
# pkill the bash runner; the EXIT trap kills the scraper service and
# writes a final INTERRUPTED report.
pkill -f soak_100_asin_24h.sh
```

## Why this exists (R-CARRY-13)

Audit `docs/scraper-multi-review-2026-05-01.md` flagged that the M9 scraper
fix stack is **code-correct but not yet load-validated**. Specifically:
the user's prior tool managed 2,000-5,000 ASIN/day with zero blocks. We
need to see whether a self-built scraper from this OCI Asia server can
reproduce that — without paying for a residential proxy.

This soak answers a narrower question: **at 100 fetches/24h pacing
(~14 min/each), with DIRECT only, do we see Amazon blocks emerge?**

When DECODO_PROXY_URL/SMARTPROXY_URL are set later, JKI-86 reruns the
same shape with proxies and we have a side-by-side to make the
breakeven decision.
